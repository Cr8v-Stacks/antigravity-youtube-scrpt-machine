# Core Workflow

## Channel Identity

Define the channel in one sentence before building content. The sentence must act as:

- Channel description.
- SEO backbone.
- Scripting filter.
- Title filter.
- Topic rejection filter.

If a video does not fit the promise, challenge it or reframe it.

## Content Engine

Prefer angle templates over broad topic clusters. Topic clusters are useful for organization, but angles create repeatability and differentiation.

Use these universal angle templates:

- System loop reveal: show a closed loop trapping behavior or outcomes.
- Invisible rule explanation: expose a rule people follow without noticing.
- Incentive dissection: explain why a system behaves as it does.
- Contradiction model: show two opposing truths shaping the story.
- Human simulation: present behavior as if humans, markets, or systems were operating under visible rules.
- Strange origin: explain how something familiar began in a surprising way.
- Hidden cost: reveal the consequence behind something people think is simple.
- Project or company tension: introduce a real actor, number, risk, and obstacle.

## Topic Ideation & Suggestion Process

Before generating a new Script Control Brief, when the user requests topic ideas or wants to validate a raw topic, the system must present a structured suggestion menu to help select the strongest angle. Do not proceed to full scripting until the topic's angle, core hook, and conflict are aligned.

When ideating or validating a topic:
1. **Suggest 3 Distinct Angles**: Map the raw topic onto three different universal angle templates from the Content Engine (e.g., a "System loop reveal" angle, a "Strange origin" angle, and a "Hidden cost" or "Project tension" angle).
2. **For each angle, propose**:
   - **Clickable Title**: A curiosity-driven packaging option.
   - **Core Hook Attack**: The specific viewer misconception or folk answer this angle will destroy in the first 15 seconds.
   - **Main Tension / Narrative Engine**: What drives the middle of the video (the conflict, grid constraint, rival, or system loop).
   - **Key Visual Scene**: A concrete, highly visual moment or artifact the viewer will see (e.g., Cueva de los Murciélagos poppy baskets or cuneiform beer ration tables) to anchor the explanation.
3. **Provide a Recommendation**: Guide the user toward the strongest option with a 1-2 sentence positioning rationale.

## Misconception Research Pass

Before writing the opening for factual, historical, science, finance, technology, system-explanation, or origin videos, identify what the viewer probably thinks the answer is. Do this before scripting, not after.

Build a quick misconception map:

1. The most common folk answer.
2. The second obvious answer or partial truth.
3. The modern assumption viewers may project backward.
4. The oversimplified school-textbook version.
5. The answer that is technically true but too late, too narrow, or incomplete.

Choose the best 2-3 targets for the opening based on commonness, speed, tension, and closeness to the real answer. Do not attack weak strawmen. Attack beliefs the viewer may actually hold.

The first 30-60 seconds should usually work like this:

1. Name the obvious answer the viewer came with.
2. Break or complicate it quickly.
3. Name a second plausible answer if it creates more curiosity.
4. Explain why that answer is also incomplete.
5. Introduce the real engine of the story without fully teaching it yet.

For "What did humans do before money?", the likely targets are barter, shells/cowries, and direct trade as a constant internal market. The stronger replacement is trust, memory, reciprocity, reputation, and delayed exchange.

For renewable-energy history, likely targets may be "this started recently," "the old invention failed because it did not work," "modern versions were the first serious versions," or "the only problem was engineering." Replace those with timing, adoption, economics, infrastructure, rivalry, or social-system reasons.

For science explainers, likely targets may be the simple classroom explanation, the intuitive physical explanation, or the wrong scale of the phenomenon. Replace those with the paradox, experiment, measurement, or hidden mechanism.

For tech reviews and product topics, likely targets may be spec-sheet assumptions, brand assumptions, hype-cycle assumptions, or "newer means better." Replace those with the real tradeoff, user constraint, or failure mode.

Do not start by asking the title question again unless it immediately leads into the wrong answer. Do not use broad imagined scenarios, salary/wage analogies, or modern admin examples when the topic has a sharper native misconception to attack.

Misconception-first does not mean every script should start with "Most people think..." Vary the attack:

- Date inversion: "Electric cars were not the sequel to gasoline cars."
- Object contradiction: "This clay tablet was doing a job coins had not been invented for yet."
- Scene contradiction: "Inside the village, nobody is swapping fish for grain."
- Viewer accusation: "The answer you were taught is too neat."
- Outcome contradiction: "The machine worked. That was not enough."

Use "Most people think..." only when it is the cleanest fastest attack, not as the default opening template.

## New Channel Framework

When the user wants to brainstorm or build a new channel, guide them conversationally instead of dumping every step at once.

Work through:

1. Lock channel identity: one sentence explaining what the channel explores, for whom, and through what lens.
2. Build content DNA: 3-4 thematic clusters with 4-6 topic examples each.
3. Choose format: editorial diagrams, cinematic documentary, comic panels, notebook sketch, or mixed visual layers.
4. Create first 10 foundation videos: mix viral curiosity with authority-building ideas.
5. Define SEO and branding: title formulas, description template, visual identity, thumbnail formula, channel image prompts, and week-one test plan.

When helping create or set up a channel, include image prompts for any generated channel assets the user will need:

- Logo/profile image.
- YouTube banner.
- Optional watermark.
- Optional default thumbnail style prompt.
- Optional recurring character or visual mascot prompt, when the channel concept benefits from one.

Ask the most useful 1-2 questions at a time:

- What topics does the user genuinely know or care about?
- Is the audience broad or niche?
- Is the angle curiosity, education, systems thinking, history, entertainment, or authority?
- Is the audience trying to understand themselves or understand the world?

Challenge generic positioning before building a content plan.

## Draft Script Handling

When the user provides a draft script, treat it as raw material, not final truth.

Before rewriting or packaging:

1. Diagnose whether the hook matches the channel's preferred style.
2. Estimate word count and runtime, then compare them to the target runtime before rewriting.
3. Flag where the script is too even, too academic, too vague, too slow, too essay-like, or off-brand.
4. Identify missing tension, missing contradiction, missing research anchors, weak payoff, missing scenes, or missing 20-30 second re-hooks.
5. Preserve strong user ideas, but rewrite weak sections instead of regurgitating them.
6. Explain the main correction briefly before giving the improved output.

## Runtime And Word Budget

Before drafting any long-form script, choose a target runtime and calculate the word budget.

Use 195 words per minute as the default narration planning speed to align with automated TTS and conversational delivery with natural pauses (approx. 3.25 words per second).

Default planning formula:

`target words = target minutes x 195`

### Shot Pacing and Narration Duration Calibration

To prevent audio/visual desync and incorrect duration estimates on the timeline:
- **Calibrate Shot Durations to Actual Word Counts**: Do not assign arbitrary timestamps without checking how many words are spoken during that shot.
- **Narrative Timing Formula**: Use the default narration speed of 195 WPM (3.25 words per second) to calculate realistic shot durations:
  `Shot Duration (seconds) = Word Count of Narration / 3.25`
- **Shot Duration Boundaries**: 
  - A 3-second shot can cover at most 9-10 words.
  - A 4-second shot can cover at most 13 words.
  - A 5-second shot can cover at most 16 words.
  - If a single sentence exceeds the duration limits, split the sentence/phrase into multiple shorter shots.
- **WPM Pacing Check**: Before delivery, verify that the total duration of the shot list matches the actual word count of the script using the formula: `Total Duration = Word Count / 3.25`.

Set a planning range before writing:

- 5 minutes: 625-700 words.
- 6 minutes: 750-840 words.
- 7 minutes: 875-980 words.
- 8 minutes / 8:00-8:30: 1,080-1,250 words.
- 9 minutes: 1,215-1,350 words.
- 10 minutes: 1,350-1,500 words.
- 12 minutes: 1,620-1,800 words.

For retention-heavy channels, prefer the lower half of a chosen range unless the research and outline genuinely need more room. A script that reaches the upper half must earn it with scenes, twists, evidence, chronology, conflict, and re-hooks, not repeated explanation.

For monetization-ready long-form videos, default to at least 8 minutes when the topic can honestly sustain it. Treat 8 minutes as the floor, not the ceiling. The common standard band is about 8:00-9:30, or roughly 1,080-1,350 words at normal documentary pace. Use 9-10+ or 10-12 minutes when the research sheet, outline, and retention map support the extra talking points. Use 6-7 minutes only when the topic is too thin for 8 minutes, the format is intentionally shorter, or the user requests it.

Do not pad to reach 8 minutes. If the concept is short, either find stronger scenes, evidence, examples, reversals, and re-hooks that genuinely earn the runtime, or recommend a shorter format.

After drafting, estimate runtime in the output:

`estimated runtime = word count / 135`

If the script is more than 10% over the chosen target, audit the cause before cutting. If the excess comes from filler, repeated explanation, jargon, soft bridges, or duplicated lists, revise it down. If the extra runtime comes from necessary proof, scenes, chronology, conflict, or payoff, update the target honestly and expand the retention map and visual plan to match.

## Script Control Brief

After research, misconception mapping, and runtime budgeting, write a Script Control Brief before drafting any substantial script. Treat it like the script's operating prompt: it should guide the output and prevent drift.

Keep it concise, but include:

1. Topic promise: the exact viewer question the video satisfies.
2. Target runtime and word budget.
3. Content type: mini-documentary, system explainer, energy history, science explainer, tech review, etc.
4. Viewer starting belief: what the viewer probably thinks before watching.
5. Opening targets: the 2-3 misconceptions or incomplete answers to attack first.
6. Core replacement idea: the truth the video gradually earns.
7. Retention engine: the main loop, such as assumption, attack, confusion, proof tease, scene, complication, reveal.
8. Scene ladder: the major visible scenes, objects, people, dates, places, or conflicts that will carry the narration.
9. Re-hook plan: the pressure point every 20-30 seconds.
10. Forbidden drift: what the script must avoid, such as essay voice, early full model reveal, padding, generic setup, or modern analogies that do not fit the topic.
11. Ending target: the emotional or philosophical landing.

Do not write the script until the brief is coherent. If the brief shows the topic cannot sustain the target runtime, say so and either deepen the research angle or recommend a shorter version.

## Opening Attack Ladder

After the Script Control Brief and before drafting, write a timestamped first-minute Opening Attack Ladder. This turns the hook idea into execution.

For long-form scripts, use 4-6 micro-beats:

1. `0:00-0:05`: start with the viewer's wrong belief, a visible contradiction, a strange object, a date inversion, or a proof clue.
2. `0:05-0:15`: attack or complicate that belief.
3. `0:15-0:30`: show a concrete proof tease, scene, number, object, person, or reason the obvious answer is incomplete.
4. `0:30-0:45`: remove the second obvious answer or deepen the question.
5. `0:45-1:00`: promise the real engine without summarizing the full explanation.

The ladder must be native to the topic. Do not import generic modern admin analogies when the topic has stronger native scenes. For a money-origin video, use barter, shells, clay tablets, grain, village trust, outsider exchange, and early coinage. For an energy project, use project scale, technology, rivals, cost, grid constraint, weather, or failure risk. For a behavior video, use a recognizable social moment, decision pressure, contradiction, or consequence.

If the ladder feels like a classroom introduction, rewrite it before drafting.

## Narrative Texture Rules

Before drafting the script, write 4-6 episode-specific texture rules that prevent article voice.

They must lock:

1. Voice texture: documentary, investigative, conversational, tense, reflective, or another chosen mode.
2. Sentence rhythm: where short pressure lines are allowed and where fuller scene paragraphs are needed.
3. Scene rule: what kind of visible moment, object, action, person, place, number, date, or conflict should appear in most paragraphs.
4. Re-hook rule: what kind of viewer update appears every 20-30 seconds.
5. Forbidden moves: exact phrases, generic bridges, weak summaries, early full-model reveals, or wrong-topic analogies to avoid.
6. Model-drip rule: what explanation must be delayed until scenes prove it.

The script should not read like an essay that happens to be spoken aloud. It should feel like the viewer is repeatedly updating a belief because each scene changes the answer.

## Script Voice Standard

This section defines the exact voice standard for all long-form factual, historical, science, origin, and system-explanation scripts. It is the difference between a script that reads like Wikipedia spoken aloud and a script that keeps a viewer watching.

The gold standard voice has these properties:

### 1. Start With The Viewer's Immediate Reality, Not A Question

The best hooks do not ask "Did ancient humans get high?" They speak directly to something the viewer is doing, thinking, or picturing right now.

- ❌ Weak: "Let's start with a picture you probably have in your head."
- ❌ Weak: "Most people think..."
- ✅ Strong: "You probably have a picture in your head right now of how this started."
- ✅ Stronger: "Right now, you're probably drunk or about to get drunk."

The viewer-first hook treats the viewer as an active participant, not a passive learner. It makes the first sentence feel personal and immediate.

### 2. Build A Chronological Escalation Engine — Never Jump Around

Every step in the timeline must flip a major assumption and push the story forward. The structure is:

`Wrong assumption → Evidence that breaks it → New assumption → Evidence that breaks that → Deeper truth`

The timeline must climb in one direction. Not topic-hopping. Not jumping from Spain to Turkey to Sumer and back. A clean chronological arrow:

`10 million years ago → 13,000 years ago → 12,000 years ago → 7,000 BCE → Egypt`

Each step should make the viewer say "wait, so that means..." before the script answers the question. Never give the conclusion before the scene earns it.

- ❌ Weak: "Humans used psychoactive plants for pain relief, social cohesion, and political alliances. Here are the examples."
- ✅ Strong: "Pain relief was only the beginning. Because about 13,000 years ago, another group was brewing in a cave near Haifa, Israel."

### 3. Keep The Re-Hook Conversational, Not Academic

Every 20-30 seconds, re-engage the viewer. But the re-hook must sound like a person talking, not a professor lecturing.

- ❌ Academic re-hook: "This demonstrates that the relationship between humans and psychoactive substances is more complex than initially assumed."
- ✅ Conversational re-hook: "Think about that. In a world without aspirin, antibiotics, or dentistry, a tooth infection could kill you."

The best re-hooks use:
- A direct question to the viewer ("What would motivate that?")
- A direct address ("Think about that.")
- A relatable comparison ("If you had to choose between a rotting tooth or using a poppy capsule, what are you picking?")
- Dry self-aware humor ("Yeah, same.")
- A scale jump that creates awe ("12,000 years ago, they were possibly building the first monuments in human history just to have a place to drink together.")

### 4. Dry, Self-Aware Humor — Woven Into Logic, Not Dropped In

The humor works because it appears at the moment the viewer is already thinking the same thing. It is not a joke. It is the script acknowledging the obvious.

- ❌ Forced: "It's quite ironic that these ancient peoples used the same substances we consider problematic today."
- ✅ Dry and earned: "Honestly, if you were working in the desert heat and someone gave you a choice between a bag of coins or 5 litres of cold beer, what are you picking? [clears throat] Yeah, same."

Use this pattern sparingly. Once or twice per script at the most. It only works when the viewer was already thinking it.

### 5. The Big Payoff Recap — Stack The Timeline, Land The Thesis

The ending should not summarize what the video covered. It should compress the timeline into a single chain of facts, then land the thesis on top like a punch.

- ❌ Weak ending: "So we can see that mind-altering substances have played an important role throughout human history."
- ✅ Strong ending (pattern): Name the time → event → time → event → time → event → then the thesis.

Example of the pattern at work:
"10 million years ago, your ancestors were drawn to fermented fruit. 13,000 years ago, they were deliberately brewing in caves before they knew how to farm. 12,000 years ago, they were possibly building the first monuments just to have a place to drink together. We didn't build civilization and discover alcohol. Alcohol might be the reason we built it at all."

The thesis should feel earned, not declared. Build to it through the recap, not before it.

### 6. Paragraph And Scene Rules

Every paragraph must pass this test: **can you see it happening?**

- If the paragraph is all explanation with no scene, object, person, date, or location — rewrite it.
- If two adjacent paragraphs make the same point — merge or cut one.
- If a paragraph starts with a thesis before showing the evidence — flip the order.

The body paragraphs should feel like the viewer is being shown evidence, not taught a lesson.

### 7. Forbidden Moves

- Starting with definitions.
- Saying "this demonstrates" or "this shows us that."
- Starting a sentence with "It is important to note that."
- Summarizing the model before scenes prove it.
- Using the phrase "in conclusion" or "in summary."
- Academic transition phrases like "Furthermore," "Additionally," "Moreover."
- Passive voice for active historical events: not "Beer was given to workers" but "Workers were paid in beer — about 4 to 5 litres a day."
- Long explanatory paragraphs with no scene, person, place, or object to anchor them.
- Starting the closing paragraph with a general truth statement instead of a stacked timeline recap.


## Retention Beat Map

Before drafting a standard long-form script, create a timestamped Retention Beat Map. This is the bridge between the Script Control Brief and the narration.

Use 20-30 second beats. Each beat must include:

1. Timestamp range.
2. Viewer question or pressure point.
3. Scene, object, person, date, place, conflict, or proof shown.
4. Mini-reveal or update to the viewer's belief.

For an 8-minute video, aim for 16-24 beats and make the map cover the full target runtime, not just the first 5-7 minutes. Do not write the full script until the beat map has enough scenes, reversals, and proof points to sustain the target runtime.

If two adjacent beats make the same point, merge or replace one. If a beat cannot be visualized, rewrite it as a scene or remove it.

## Script Structure

For broad long-form videos, use:

1. Hook: pattern interrupt, contradiction, strange question, or uncomfortable truth.
2. Setup: why the question matters.
3. Simple model: 2-3 layers, explained clearly.
4. System reveal: hidden loop, incentive, history, or mechanism.
5. Modern manifestation: how it appears now.
6. Final twist: philosophical, unsettling, or clarifying implication.

For retention-heavy mini-documentary and origin videos, use:

1. Assumption attack: break the answer the viewer brought into the video.
2. Proof tease: show one concrete reason the common answer is incomplete.
3. Scene one: put the viewer into the first visible moment, object, person, or social situation.
4. Escalation: add a date, place, object, conflict, or scale jump.
5. System reveal: let the model emerge from scenes rather than stating it early.
6. Second misconception break: correct the next plausible but incomplete answer.
7. Payoff: explain the deeper mechanism.
8. Reflection: connect the mechanism to the modern world or human behavior.

For Shorts, use:

1. Hook: 0-5 seconds, immediate curiosity.
2. Setup: introduce person, company, technology, project, event, or behavior.
3. Tension: explain why it matters.
4. Payoff: reveal outcome or answer.
5. Ending: connect to present day or the wider system.

Do not write Shorts like smooth articles. Use sharper contrast, faster pivots, and stacked "wait, what?" beats. If the script explains too evenly, tighten it until each section either raises curiosity, creates conflict, or answers a question.

For fact-driven history Shorts, stack 2-3 shocks in the first 10-15 seconds:

- Normal expectation vs strange fact.
- Weird number or impossible scale.
- Surprising real-world result or duration.

Then introduce the central question early: if this worked, why did it not become the future?

For Shorts, keep the ending concrete. Do not drift into abstract reflection when a sharper payoff is available. The final third should answer:

- What was the surprising outcome?
- What blocked adoption, scale, or public acceptance?
- Who was competing, threatened, ignored, or proven wrong?
- Why does this specific story matter now?

If the ending feels like "the future arrived early" without a concrete obstacle, rival, consequence, or twist, rewrite it.

Compress the ending. Shorts usually land better with one sharp explanation than with several explanatory sentences. Prefer:

- "It worked. It just did not fit the system being built around it."
- "The invention was real. The market moved somewhere else."
- "The machine proved the idea. The industry chose a different path."

Use these as patterns, not fixed lines.

## Content-Type Opening Rules

Different content types need different openings. Do not use one generic explainer intro across every niche.

For history mini-documentaries, origin stories, ancient history, economics history, and "what happened before X?" videos, the opening should attack the most common wrong answer before explaining the correct answer. The first minute should feel like a mystery: "Everything you think about this topic is probably wrong, so what actually happened?" Delay the answer long enough to create tension. Destroy the obvious answers first, then create demand for the real answer.

For renewable-energy project videos, open with tension, scale, risk, money, competition, and possible failure. The viewer should quickly know what was promised, how big the project is, what could go wrong, and who is threatened or competing.

For technology replacement videos, open with future promise, disruption, and the incumbent being challenged. The viewer should quickly understand what dominant technology may be replaced, who is betting against it, and why the replacement is not guaranteed.

For business or company stories, open with a decision, market bet, collapse risk, hidden cost, or competitive pressure. Do not begin with a company biography unless the person's origin story is itself the curiosity object.

For science explainers, open with a paradox, impossible-sounding result, strange measurement, or real experiment. Do not begin with definitions. Let the viewer ask how the result can be true.

For human behavior and psychology videos, open with a recognizable behavior that feels uncomfortable or contradictory. The hook should make the viewer think, "I have done this," or "I have seen people do this," before naming the concept.

For system-explanation videos, open with the visible outcome that seems irrational, unfair, inefficient, or self-defeating. Then reveal the hidden incentive loop that makes the outcome predictable.

For invention-history videos, open with the strongest contradiction: the invention worked but failed, arrived early but disappeared, solved a problem but was rejected, or was beaten by a worse-looking alternative.

For finance, money, and economic-system videos, open with the mistaken folk explanation. Money topics should attack the naive barter story, the "people just traded" assumption, or the idea that modern systems are natural and inevitable.

For future-trend videos, open with the current consensus being challenged by a real signal: a company investment, research breakthrough, policy shift, supply bottleneck, cost curve, or infrastructure constraint.

For mystery or forgotten-history videos, open with an absence, contradiction, missing record, strange survival, or overlooked object. The first minute should make the viewer feel there is a hidden story to uncover.

## Hook Rules

Never start with definitions, background, or teaching. Start with:

- Conflict.
- Surprise.
- Contradiction.
- Mystery.
- Consequence.
- A specific person, company, date, place, or number.
- The strongest curiosity object in the story.

- **The Direct, Conversational "You" Hook**: Speak directly to the viewer's immediate physical reality or immediate mental state.
  - *Viewer Mental State Hook*: "You probably have a picture in your head right now of [concept/historical scene]..." (This is highly effective because it directly targets and updates the viewer's current imagination/assumptions).
  - *Viewer Physical Reality Hook*: "Right now, you're probably [action, e.g., drunk or about to get drunk]..." (This immediately grounds the viewer in a recognizable personal situation).

Bad: "Let's start with a picture you probably have in your head." (Too passive/instructional; feels like a lecture beginning).
Better: "You probably have a picture in your head right now of..." (Directly engages the viewer's current thoughts).
Bad: "This is the first [technology]."
Better: "The first [technology] worked. The world still rejected it."

Bad: "Humans follow crowds because of social psychology."

Better: "Humans trust crowds even when the crowd is clearly wrong."

When choosing between a broad label and a concrete surprise, use the concrete surprise. A title built around a weird number or unexpected capability is usually stronger than a title built around the broad category because it creates an immediate comparison in the viewer's mind.

The hook should attack the most common wrong answer, not merely state the correct answer.

Bad: "What did humans do before money?"

Better: "Most people think the answer is barter. The evidence suggests something stranger."

Bad: "Can renewable energy power AI?"

Better: "Most people think AI's biggest energy problem is computing power. It isn't."

Bad: "Who built the first wind turbine?"

Better: "Most people think modern wind power started in the 1970s. The real story starts almost a century earlier."

Do not spend the first 30 seconds explaining why the topic matters. The viewer already clicked because the topic matters enough. The first 30 seconds should answer: "Why is what I think wrong?" or "What is the mystery I need solved?"

For curiosity-driven scripts, the opening should often eliminate two obvious answers before offering the real one. This traps curiosity because the viewer now needs the third answer.

Example pattern:

1. State the question.
2. Name the obvious answer.
3. Challenge it with evidence or contradiction.
4. Name the second likely answer.
5. Explain why that is incomplete or too late.
6. Ask the deeper question.
7. Promise the real answer without revealing it too early.

For mini-documentary hooks, prefer misconception, mystery, hidden truth, forgotten fact, or challenge-assumption openings over broad importance openings.

## Sticky Number Rule

For history, science, technology, energy, business, and finance Shorts, find the weirdest memorable number and build the packaging around it when possible.

Humans remember:

- Unusual quantities.
- Surprising dates.
- Odd measurements.
- Unexpected durations.
- Strange efficiency numbers.
- Extreme costs or scale.
- Time gaps between invention and adoption.

They do not remember textbook phrasing such as "an early technology helped demonstrate future potential."

Use the number as:

- Title anchor.
- Thumbnail text.
- First 5-second hook.
- Retention beat later in the script.

Do not repeat the same sticky number in the same way. Use it as a progression:

1. Hook: reveal the number as the curiosity object.
2. Setup: attach the number to a person, place, project, or event.
3. Tension: explain why the number is strange, unlikely, rejected, expensive, early, inefficient, or hard to believe.
4. Payoff: show what the number proves or changes.

If the hook says "[weird number] + [surprising outcome]," the later beat should not say the same thing again. It should add a new layer, such as:

- Why the number was technically difficult.
- Why the result lasted longer than expected.
- Why nobody copied it despite the proof.
- What limitation prevented adoption.
- What modern comparison makes the old number feel shocking.

Title ladder pattern:

- Weak: "The First [Technology]."
- Better: "The [Technology] That [Surprising Outcome]."
- Strongest: "The [Technology] With [Weird Number Or Time Gap]."

## Retention Rules

- **Chronological Escalation**: Set a clear chronological starting point (e.g., millions of years ago, or a specific historical date) and move strictly forward, letting each step build tension or flip a major assumption. Avoid jumping back and forth across timelines.
- **Relatable Human Transitions**: Integrate self-aware, conversational humor or rhetorical check-ins with the viewer (e.g. direct questions or dry, self-deprecating commentary like "Yeah, same").
- **Watching Experience over Reading**: Never write paragraphs that summarize lessons or explain abstract models before scenes prove them. Ground abstract claims immediately in tangible actions, real people, specific tools, dates, or locations.
- **Re-hook Progression**: Every 20-30 seconds, instead of introducing disconnected facts, build a logical chain: assumption -> attack -> concrete proof tease -> structural twist -> reward.

Every 20-30 seconds, reset viewer interest with a new pressure point:

- Question.
- Fact.
- Date.
- Number.
- Statistic.
- Company.
- Location.
- Historical event.
- Comparison.
- Risk.
- Opportunity.
- Surprise.
- Problem.
- Consequence.
- Emotion.
- Visual payoff.
- Misconception break.
- Mini-reversal.
- Proof tease.
- Scale jump.

Do not spend multiple minutes circling the same point.

For any script type, do not let the narration become pure explanation for too long. Every 15-30 seconds should ideally give the viewer a concrete reward. The reward depends on the content type:

- Factual/history/science: number, date, name, place, object, experiment, discovery, measurement, source detail, or contradiction.
- Business/technology/energy: company, cost, capacity, timeline, competitor, bottleneck, market signal, failure point, or adoption obstacle.
- Human behavior/psychology: familiar scenario, social consequence, internal contradiction, relatable example, decision pressure, or "I have done this" moment.
- Systems/explainers: visible outcome, hidden rule, incentive loop, constraint, exception, tradeoff, or real-world consequence.
- Story/documentary: character decision, emotional turn, reveal, reversal, mystery clue, location change, or new stake.
- Visual/tutorial/process content: step result, before-and-after, mistake, tool choice, constraint, comparison, or practical payoff.

Explanation is allowed, but it must be interrupted by viewer rewards. A good script should feel like question, answer, new question, answer, new question, not question followed by a long lecture.

When a section makes an abstract claim, immediately ground it in a tangible example from the topic's own world. If the script is about behavior, show a person making the mistake. If it is about a company, show the decision or market pressure. If it is about a technology, show the constraint or test. If it is about history, show the object, place, date, or social situation.

If a viewer could say "okay, I get it" before the section ends, add a new fact, example, complication, or question.

Do not reveal the full explanatory model too early. Drip the model through scenes, proof, contradiction, and escalation. Avoid early summary lines such as "Before money, humans relied on trust, memory, and reciprocity" unless the viewer has already seen a concrete moment that makes the summary feel earned.

Use this retention engine for mini-documentary scripts:

1. Assumption.
2. Attack.
3. Confusion.
4. Proof tease.
5. Scene.
6. Complication.
7. New question.
8. Reveal.

Do not write a chain of truth statements. Write a chain of viewer updates.

## Research Rules

Use browsing or verified sources when the request depends on current or factual details such as companies, statistics, prices, new projects, regulations, release dates, or recent events.

Factual scripts should contain:

- Real companies.
- Real projects.
- Real numbers.
- Real timelines.
- Real locations.
- Specific comparisons.

Specifics create authority and retention.

For factual content outputs, include a "Research Material" section when the user asks for upload-ready or production-ready work. Provide source links and the key facts used so the user can cross-check the script.

Before factual or historical scripts, gather the research foundation and misconception map first. Use them to choose the hook, title, conflict, opening targets, and details instead of writing a generic script and adding facts afterward.

Research must look for wow-factor context, not just object facts. For historical topics, check for:

- Rivals and competitors.
- Public rejection or misunderstanding.
- Business failure or market timing.
- Patent, company, merger, or industry consequences.
- Famous names connected to the story.
- A detail that changes how the viewer understands the main person or machine.

Use these details only when they tighten the story. Do not add famous names as trivia unless they create tension or a clearer payoff.

## Paragraph Rule

Every paragraph should contain enough connected information to justify a visual sequence.

Every paragraph must pass the scene test: "Can I see this happening?" If the answer is no, rewrite the paragraph around a person, object, event, setting, action, conflict, or visible system. Abstract conclusions can appear, but only after a concrete moment earns them.

Avoid many isolated short sentences unless writing a deliberate punchy hook. For body sections, give editors substance to cut around: actor, action, reason, tension, and visual cue.

Avoid "truth paragraphs" that only state the lesson:

- Weak: "Before money, humans relied on trust, memory, and reciprocity."
- Stronger: "If someone shared meat after a hunt, nobody pulled out a price list. The village remembered who had shared, who had taken, and who would be expected to help when the dry season came."

The body should feel like scenes creating ideas, not ideas looking for scenes afterward.

## Production Mindset

Optimize for speed, consistency, and retention before polish.

Default production pipeline — all gates are mandatory in order:

1. **Research** — misconception map, wow-factor context, verified sources.
2. **Script Control Brief** — lock angle, runtime, word budget, opening targets, forbidden drift before writing.
3. **Opening Attack Ladder** — map the first-minute micro-beats before writing.
4. **Narrative Texture Rules** — lock voice, rhythm, scene rule, re-hook type, forbidden phrases.
5. **[Read `references/core-workflow.md` § Script Voice Standard before writing]** — internalise the viewer-first hook, chronological escalation engine, conversational re-hooks, earned humor, stacked-timeline ending, and forbidden moves.
6. **Write script** — watching experience, not reading experience.
7. **Script Voice Self-Audit Gate** — critique the draft against the Voice Standard before moving on. See § Script Voice Self-Audit Gate below.
8. **Fact-Check Gate** — sweep all specific claims before shot-by-shot. See § Fact-Check Gate below.
9. **Shot-by-shot image prompts** — timed, standalone, director-minded visual beats.
10. **Run validators** — `script_quality_lint.py` then `production_pack_lint.py`.
11. **Package metadata** — SEO description (250-400 words), hashtags, tags, research material.
12. **User production** — voiceover, assembly, subtitles, sound, upload.

The idea carries the video. Do not overcomplicate production.

## Script Voice Self-Audit Gate

Run this gate immediately after writing the script and before the Fact-Check Gate or shot-by-shot work. It is a fast, structured self-critique that confirms the script sounds like the channel, not like a Wikipedia article.

The gate asks seven questions. Each must get a YES before proceeding.

1. **Hook question**: Does the first sentence speak directly to something the viewer is doing, thinking, or picturing right now — not a question, not a definition, not "most people think"?
2. **Chronological engine**: Does the timeline move in a single forward direction, with each step flipping an assumption before introducing the next? Or does it jump between locations, topics, or eras without a clear escalation arrow?
3. **Scene test**: Sample three paragraphs from the body. Does each one contain at least one concrete person, place, object, date, action, or artifact the viewer can picture? Or are they explanation paragraphs with nothing to see?
4. **Re-hook test**: Can you find a conversational re-hook (a direct question, a direct address, or a relatable "yeah, same" moment) within every 30-second window? Or are the transitions academic bridges ("this demonstrates," "furthermore," "it is worth noting")?
5. **Forbidden move scan**: Search for these phrases. If any are present, rewrite the sentence.
   - "This demonstrates"
   - "This shows us that"
   - "It is important to note"
   - "Furthermore" / "Additionally" / "Moreover"
   - "In conclusion" / "In summary"
   - "As we can see"
   - Any passive construction for an active historical event (e.g. "beer was given to workers" → "workers were paid in beer")
6. **Ending test**: Does the ending stack the timeline (time → event → time → event → thesis) rather than summarising what the video covered? Does the thesis land on top of the stack like a punch, not before it?
7. **Article voice test**: Read the script aloud in your head. Does it sound like a person telling you something extraordinary — or like a textbook section read into a microphone?

### Gate Output Format

Report the audit before proceeding:

```
## Script Voice Self-Audit

1. Hook — YES / NO — [note if NO]
2. Chronological engine — YES / NO — [note if NO]
3. Scene test — YES / NO — [sampled paragraphs: lines X, Y, Z]
4. Re-hook test — YES / NO — [note weak window if NO]
5. Forbidden moves — CLEAN / FOUND — [list any found]
6. Ending — YES / NO — [note if NO]
7. Article voice — PASS / FAIL — [note if FAIL]

Gate: PASS / FAIL
```

If any item is NO or FAIL, fix the script before proceeding. Do not note a failure and continue anyway. A voice audit failure means the script will regress to academic essay output when it reaches the shot-by-shot stage.

## Fact-Check Gate

For factual, historical, science, archaeology, economics, or origin videos, run a fact-check sweep of the completed script before beginning shot-by-shot image prompt work.

This is mandatory because script corrections are fast, but image prompt corrections are expensive. An error that passes into shot-by-shot work gets locked into timestamps, narration line references, and prompt copy.

The gate runs in four steps:

1. Extract all claims containing a specific number, date, mechanism, attributed artifact, biological claim, or behavioral/institutional attribution.
2. Flag each claim as ✅ Verified, ⚠️ Hedged (contested but supportable), or ❌ Inaccurate (wrong or misleading).
3. Correct all ❌ claims and soften all ⚠️ claims using the correction principles in `references/fact-check.md`.
4. Report results in the gate output format before proceeding to shot-by-shot work.

Do not skip the gate and note the failure. Fix before proceeding. The gate passes only when all ❌ items are corrected and all ⚠️ items are appropriately hedged.

Common failure patterns by content type:

- **History / Archaeology**: Overstating certainty of excavation evidence, collapsing evolutionary timelines into a single wrong date, attributing diffuse cultural practices to a single site.
- **Science / Biology**: Mapping receptor biology onto oversimplified analogies, using the wrong timeline for evolutionary events, presenting one model of a disputed mechanism as settled.
- **Economics / Finance**: Describing ancient economies with modern economic vocabulary that did not exist as concepts, attributing intentionality when the evidence only shows correlation.
- **Technology / Energy**: Conflating feasibility with viability, conflating proof-of-concept dates with deployment dates.

For the full correction principles, failure pattern catalogue, and gate output format, read `references/fact-check.md`.

## Pre-Delivery Self-Audit

Before delivering a substantial script or production pack, run a final self-audit in this order:

1. Runtime check: confirm target runtime, target word range, actual or estimated word count, and estimated runtime.
2. Opening check: confirm the first 5-10 seconds attack the viewer's likely belief, not the topic in general.
3. Opening ladder check: confirm the first minute has 4-6 micro-beats and does not collapse into a calm explanation.
4. Model-drip check: confirm the full explanation is not summarized before scenes prove it.
5. Scene check: sample body paragraphs and ask, "Can I see this happening?"
6. Voice audit check: confirm the Script Voice Self-Audit Gate was run and passed. If not, run it now.
6a. Article-voice check: remove generic essay phrases, textbook bridges, repeated abstract summaries, and robotic cadence.
7. Re-hook check: confirm every 20-30 seconds has a pressure point: reversal, proof tease, scene change, question, number, date, object, conflict, or scale jump.
8. Native-context check: confirm the script uses the topic's own world, not generic modern analogies or unrelated admin examples.
9. Cut check: remove repeated truth statements, soft bridges, and explanatory padding.
10. Delivery check: confirm the output includes the Script Control Brief, Opening Attack Ladder, Narrative Texture Rules, and the script follows them.
11. Fact-check gate check: confirm the Fact-Check Gate was run and passed before shot-by-shot work began. If the pack includes shot-by-shot prompts and no gate was run, run it now and apply any corrections before final delivery.
12. Visual pack check, when image prompts are included: confirm the selected visual system, character prompt style, prompt depth, shot count, timestamp pacing, narration coverage, and prompt splitting (shots ≤3 seconds = 1 prompt; shots 5-8 seconds = 2-3 individual prompts where narration content warrants it).

If a file or draft text is available, run `scripts/script_quality_lint.py` as a mechanical tripwire. If a full production pack includes image prompts, also run `scripts/production_pack_lint.py`. Fix failures before delivering. Passing the scripts does not replace the self-audit.

Use the right gate for the artifact:

- Script flow/opening test: use a small script-body range, but still require `--min-beats 16 --target-duration-sec 480` so the flow proves it can sustain 8 minutes.
- Finished standard long-form script: use `--preset standard-8` when the target is in the 8:00-9:30 band. For intentionally longer scripts, set an honest longer target and apply the same retention audit manually.
- Finished standard long-form full production pack: run both `scripts/script_quality_lint.py --preset standard-8 <file>` and `scripts/production_pack_lint.py --preset standard-8 <file>` when the target is in the 8:00-9:30 band. For intentionally longer packs, scale shot count and timestamps beyond the standard preset.

Do not call a script flow "production-ready" until the finished script body also passes the word-count gate and the editorial self-audit.
Do not call a production pack "shot-by-shot" until the visual prompt gate passes or the pack is clearly labeled as a partial shot-list draft.
